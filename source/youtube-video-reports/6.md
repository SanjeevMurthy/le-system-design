The Architect’s Handbook: Comprehensive Notes on Modern System Design

In the high-stakes environment of technical interviews at firms like Meta and Amazon, the distinction between a "hire" and a "strong hire" is determined by an engineer's ability to navigate ambiguity using a disciplined, first-principles methodology. System design is not an exercise in buzzword bingo; it is the art of making defensible engineering trade-offs under constraint. A repeatable framework is the architect’s primary defense against "premature optimization"—the fatal error of solving for billion-user scale before the core functional contract is satisfied. By adhering to a structured delivery model, we ensure that the "minimal working system" is robust before proactively identifying and blunting the bottlenecks of global distribution.

1. The Hello Interview Delivery Framework

A senior-level design session must move with surgical precision from ambiguity to technical implementation. This framework prevents the candidate from getting "lost in the weeds" and ensures all stakeholders understand the strategic direction of the architecture.

1.1 Requirement Clarification

Before a single component is drawn, the boundaries of the system must be strictly defined.

- Functional Requirements: These are the user-facing actions—the "verbs" of the system. For a News Feed, this includes creating posts, following users, and viewing a feed.
- Non-Functional Requirements: These are the system qualities—the "adjectives." This includes defining latency targets (e.g., p99 < 500ms), availability (e.g., "four nines"), and the CAP theorem trade-off (e.g., favoring Availability over Consistency for social media).

  1.2 Core Entities and API Design

Identify the "nouns" of the system (e.g., User, Post, Follower) to establish the data model. These entities inform the API contract—the formal agreement between client and server. Whether utilizing RESTful conventions for standard actions or WebSockets for stateful, bi-directional communication, the API must be exhaustive enough to satisfy all functional requirements.

1.3 High-Level Design (HLD) vs. Deep Dives

The HLD phase is about building a simple, end-to-end working system. It may have "warts," but it must prove the concept. Once established, we transition to the Deep Dive, where we proactively hunt for bottlenecks. This is where we address the "Fanout Problem" or resolve "Hot Shards" through advanced sharding and caching strategies.

1.4 Strategic Estimates: Math as a Design Tool

Back-of-the-envelope math is not a perfunctory ritual; it is a tool for making design pivots. For example, in a Facebook Post Search scenario:

- Scale: 1 billion posts/day × 365 days × 10 years = 3.6 trillion posts.
- Storage: 3.6 trillion posts × 1 KB/post = 3.6 Petabytes. This 3.6PB figure immediately invalidates any standard relational database approach and dictates the move toward distributed blob storage (like S3) and a highly sharded inverted index.

2. Core Architectural Concepts and Distributed Patterns

Distributed systems are volatile. To build at global scale, an architect must master patterns that maintain performance even when individual nodes fail or viral events create tidal waves of traffic.

2.1 The Fanout Problem (Read vs. Write)

In social graphs, the "Fanout" describes the delivery of a single post to millions of followers.

Feature Fanout on Load (Pull/Read) Fanout on Write (Push/Pre-computed)
Mechanism Feeds generated at request time by querying all follows. Feeds are pre-computed and stored in an "inbox" at write time.
Read Latency High (Heavy aggregation/merge at runtime). Ultra-Low (Simple O(1) read from pre-computed table).
Write Latency Low (Single write to the Post table). High (A "write thunderstorm" for large followings).
Best For "Celebrity" accounts (High fanout). "Normal" users (Small social circles).

2.2 Addressing the "Hot Shard" / "Celebrity" Problem

When a "celebrity" node (e.g., Taylor Swift) receives millions of hits, it creates a "hot key" that can overwhelm a single physical host.

- Sharding with Random Suffixes: To distribute load, append a random suffix (0 to N) to the partition key, effectively splitting one hot record into N records across different shards.
- The N-Cache Strategy: A common mistake is sharding the cache, which keeps the hot key on one node. Instead, use multiple cache instances that are not replicas of one another. The service randomly chooses one of N instances. This ensures that while we have N requests hitting the DB, those N requests are distributed across the cache layer, blunting the tidal wave without creating a new hot node.

  2.3 Reliability: SQS vs. Kafka

For fault tolerance, Amazon SQS is often superior to Kafka for specific use cases like a Web Crawler. SQS supports Visibility Timeouts and Exponential Backoff out of the box. If a crawler fails to fetch a URL, SQS hides the message and retries after an increasing delay (30s, 2m, 5m), eventually moving it to a Dead Letter Queue (DLQ) for inspection. Kafka requires manual implementation of retry topics and logic to achieve this same level of sophisticated backoff.

2.4 Consistency Models

We pivot between Eventual Consistency (Search, News Feed) where high availability is paramount, and Strong Consistency (Banking, Inventory) where the system must reflect the exact same state to all users simultaneously.

3. Data Modeling and Storage Strategy: The DynamoDB Deep Dive

The choice of database is a strategic pivot. While SQL offers rigid consistency, NoSQL solutions like DynamoDB provide the horizontal scaling required for trillions of items.

3.1 Primary Key Architecture

- Partition Key (PK): Determines the physical node location via Consistent Hashing. This ensures even distribution across the cluster.
- Sort Key (SK): Organized as a B-Tree index within the partition. This enables efficient range queries (e.g., retrieving the "latest 25 messages" in a chat) because the data is physically sorted on disk.

  3.2 Advanced Indexing (GSI vs. LSI)

- Global Secondary Index (GSI): Allows a completely different PK and SK. Essential for queries like "Find all posts by User X" when the main table is partitioned by PostID.
- Local Secondary Index (LSI): Shares the same PK but offers a different SK. Used to provide multiple sorting views (e.g., sorting by "Time" vs. "Number of Attachments") on the same data partition.

  3.3 Performance and Transactions

- DAX (DynamoDB Accelerator): A write-through cache providing microsecond latency.
- DynamoDB Streams: Enables Change Data Capture (CDC) to trigger downstream events (e.g., updating an ElasticSearch index when a post is created).
- ACID Transactions: Since 2018, DynamoDB supports atomic transactions across up to 100 items, bridging the gap between NoSQL scale and SQL reliability.

4. Detailed System Breakdowns and Real-World Scenarios

4.1 Facebook News Feed (The Hybrid Approach)

Facebook employs a hybrid model governed by a "pre-computed flag" in the follow table. For "normal" users, the system uses Fanout on Write to pre-generate feeds. For "mega-accounts," the system switches to Fanout on Read at request time to avoid the "write thunderstorm" of pushing a celebrity's post to 100 million inboxes.

4.2 Facebook Post Search (Storage at Scale)

Search utilizes an Inverted Index (mapping keywords to Post IDs). To manage the 3.6PB of storage required for a decade of posts, we use Count-Min Sketch to identify "hot" search terms for in-memory indexing, while evicting "cold" terms to S3.

4.3 WhatsApp/Messenger (Stateful Connectivity)

Messaging requires persistent WebSockets and Redis Pub/Sub for routing. To ensure "At-Least-Once" delivery, the Inbox Pattern is used: the server stores a message in the recipient's inbox and only deletes it upon receiving an Acknowledgement (ACK) from the device.

4.4 Web Crawler (The Data Pipeline)

To crawl 10 billion pages in 5 days, we must calculate the required bandwidth. A 400 Gbps instance handles ~25,000 pages/sec theoretically, but with ~30% real-world efficiency (DNS, rate limits), we hit ~10k pages/sec.

- Math: 10B \text{ pages} \div 10k \text{ pages/sec} \approx 1M \text{ seconds} \approx 11 \text{ days}. To finish in 5 days, we need \sim 4 Superduty Network Optimized instances.
- Politeness: Enforced via robots.txt and domain-level rate limiting in Redis.

  4.5 Ad Click Aggregator (Hybrid Stream/Batch)

Handling millions of clicks requires a dual architecture.

- Kappa/Lambda: Flink handles 1-minute window aggregations for real-time feedback, while Spark performs periodic reconciliation for 100% data integrity.
- Logarithmic Counting: To reduce write thunderstorms, we only update the search index when engagement crosses a power of two (2^n). This reduces write volume by orders of magnitude while maintaining "mostly correct" ranking.

5. Summary of Analytical Trade-offs for the Architect

5.1 Latency vs. Accuracy

Senior architects accept approximate data to gain massive performance.

1. Bloom Filters: Check for crawled URLs without a 3.6PB DB hit.
2. Count-Min Sketch: Approximate counts for viral search terms.
3. Logarithmic Counting: Approximating "likes" to protect the persistence layer.

5.2 Memory vs. Disk (Closet vs. Filing Cabinet)

- Redis (The Closet): Expensive, volatile, but provides microsecond access to "hot" data.
- S3 (The Filing Cabinet): Cheap, infinite, but requires a "trip down the hall" (higher latency) for "cold" data.

  5.3 Practicality vs. Purity

Academic perfection is the enemy of a working system. A senior leader knows when to "throw money at the problem"—such as hiring multiple DNS providers to bypass rate limits or using managed services like SQS—rather than building brittle, custom infrastructure. Every solution creates a new bottleneck. The architect's job is to ensure that new bottleneck is one the business can afford.

Design is iterative: identify the bottleneck, solve, and repeat.
